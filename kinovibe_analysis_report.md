# Отчет по анализу KinoVibe

## Обзор системы

KinoVibe — это сервис, предназначенный для поиска и агрегации ссылок на фильмы и сериалы с различных платформ, с акцентом на русскоязычную аудиторию. Система использует AI для улучшения поисковых запросов и рекомендаций, а также множество внешних API и инструментов для сбора метаданных и прямых ссылок на видеопотоки.

**Расположение:** `/var/www/kinovibe`
**Основной бэкенд:** `/var/www/kinovibe/backend` (порт 8110)

## 1. Методы поиска и их вызов

KinoVibe использует многоступенчатый подход к поиску, комбинируя AI, метаданные из внешних баз данных и прямые поисковые запросы к видеохостингам.

### 1.1. AI Query Refiner (`ai_query_refiner`)
*   **Назначение:** Преобразование свободного пользовательского ввода в оптимизированный поисковый запрос, учитывающий настроение, жанр, язык (приоритет русской озвучки) и популярность.
*   **Технология:** Использует модель **Google Gemini-2.5-flash** через прямой HTTP POST запрос к `generativelanguage.googleapis.com`.
*   **Логика:**
    *   `SYSTEM_PROMPT` определяет роль AI как эксперта по кино для русскоязычной аудитории.
    *   Добавляет суффиксы к запросам для YouTube (например, "русская озвучка full movie").
    *   Обрабатывает фильтры популярности ("rare", "mid", "mainstream") для формирования подсказок Gemini.
*   **Вызов:** Автоматически вызывается функцией `execute_search` по умолчанию (`mode="mood"`), если запрос не является прямым названием фильма (определяется `_is_title_query`).

### 1.2. Система рекомендаций (`get_recommendations`)
*   **Назначение:** Предоставление списка рекомендаций фильмов на основе свободного запроса пользователя.
*   **Технология:** Также использует **Google Gemini-2.5-flash**, но с `RECOMMEND_PROMPT` и активированным `Google Search grounding` (позволяет Gemini использовать поиск в интернете).
*   **Логика:**
    *   Активируется, если пользовательский запрос содержит триггерные слова (`RECOMMEND_TRIGGERS`, например, "посоветуй", "что посмотреть").
    *   Gemini генерирует список из 6-8 фильмов с названиями, годами и причинами рекомендации.
    *   Полученные фильмы обогащаются метаданными из TMDB (постер, рейтинг, описание, жанры).
*   **Вызов:** Вызывается из `execute_search`, если обнаружен триггер рекомендации.

### 1.3. Поиск по названию через TMDB (`search_by_title_tmdb`)
*   **Назначение:** Основной метод для прямого поиска фильмов и сериалов по названию, когда AI-уточнение не требуется или пропущено.
*   **Технология:** Использует **TMDB API** (The Movie Database) для параллельного поиска фильмов (`tmdb_search_movie`) и сериалов (`tmdb_search_tv`).
*   **Логика:**
    *   Ищет фильмы и сериалы, возвращая до 6 результатов для каждого типа.
    *   Результаты объединяются и сортируются по релевантности (`_tmdb_relevance` учитывает точное совпадение, начало строки, наличие вхождения и популярность/рейтинг).
    *   Возвращает "карточки" фильмов с метаданными TMDB, но **без прямых ссылок на стриминг** (они загружаются лениво по запросу к `/film/streams`).
*   **Вызов:** Вызывается из `execute_search`, когда `mode="search"` и `platform="all"`.

### 1.4. Агрегированный поиск по провайдерам (`aggregate_search`)
*   **Назначение:** Поиск контента (видео, торренты) напрямую у зарегистрированных провайдеров.
*   **Технология:** Асинхронно опрашивает множество специализированных провайдеров (YouTube, VK, RuTube, KodiK, HDRezka, Filmix, NovaTorrent, YTS) через их внутренние методы `search()`.
*   **Логика:**
    *   Принимает запрос, категорию и опциональный фильтр по `platform`.
    *   Параллельно запускает поисковые методы всех активных провайдеров с помощью `asyncio.gather`.
    *   Собирает и возвращает все найденные `SearchResult` объекты.
*   **Вызов:**
    *   Используется как запасной вариант, если `search_by_title_tmdb` не дает результатов.
    *   Вызывается для прямого поиска по конкретной платформе (например, `platform="youtube"`).
    *   Является ключевым компонентом для `find_film_streams`.

### 1.5. Поиск потоков (`find_film_streams`)
*   **Назначение:** Получение прямых ссылок на стриминг для конкретного фильма/сериала.
*   **Технология:** Вызывает `aggregate_search` с `platform="all"` для получения ссылок от всех доступных провайдеров.
*   **Логика:**
    *   Формирует запрос, используя название и год фильма.
    *   Собирает все `SearchResult` от провайдеров.
    *   Сортирует результаты по заданному приоритету провайдеров (например, `torrent` > `yts` > `vk` > `rutube` > `kodik` > `hdrezka` > `filmix` > `youtube`).
    *   Удаляет дубликаты URL-ов.
    *   Возвращает до 8 уникальных ссылок на потоки.
*   **Вызов:** Загружается "лениво" по запросу фронтенда, когда пользователь хочет посмотреть фильм.

## 2. Как ресурсы задействованы в поисках фильмов

В KinoVibe задействован широкий спектр внутренних и внешних ресурсов для обеспечения функциональности поиска и агрегации:

*   **API Ключи:**
    *   Управляются централизованно через `core.key_pool`. Эта система позволяет использовать различные ключи для одного и того же сервиса (например, несколько ключей Gemini) и отслеживать их работоспособность, предотвращая блокировку из-за исчерпания квот или ошибок.
*   **Генеративные AI (Google Gemini, Groq):**
    *   **Google Gemini-2.5-flash:** Используется для сложной обработки естественного языка: уточнения поисковых запросов (перефразирование, определение настроения/жанра), а также для генерации рекомендаций фильмов (с функцией `Google Search grounding`).
    *   **Groq (llama-3.1-8b-instant):** Используется как резервный источник для генерации кратких описаний фильмов, если TMDB или Kinopoisk не предоставили эту информацию.
*   **Базы данных метаданных:**
    *   **TMDB API (The Movie Database):** Основной источник метаданных о фильмах и сериалах: названия (включая оригинальные), постеры, описания, рейтинги, годы выпуска, жанры, длительность.
    *   **Kinopoisk.dev API:** Дополнительный источник метаданных, ориентированный на русскоязычную аудиторию, предоставляет информацию о фильмах с Кинопоиска (постеры, описания, рейтинги, годы).
*   **Инструменты для работы с видеохостингами:**
    *   **`yt-dlp`:** Внешняя утилита командной строки, активно используемая `YouTubeProvider` для:
        *   Поиска видео на YouTube.
        *   Извлечения прямых ссылок на видеопотоки (stream_url) и их протоколы.
*   **HTTP-клиенты:**
    *   **`httpx.AsyncClient`:** Асинхронный HTTP-клиент, используемый для всех запросов к внешним API (Gemini, TMDB, Kinopoisk, Groq и, вероятно, к API провайдеров стриминга). Он обеспечивает параллелизм и таймауты.
*   **Подпроцессы:**
    *   Модуль `subprocess` в Python используется для запуска внешних команд, таких как `yt-dlp`. Это позволяет интегрировать мощные сторонние инструменты.
*   **Внутреннее кэширование:**
    *   **`SearchCache` (LRU-кеш в памяти с TTL):** Хранит результаты предыдущих поисковых запросов в течение короткого времени (300 секунд). Это значительно уменьшает нагрузку на внешние API и AI-модели при повторных или похожих запросах, ускоряя отклик.
*   **Провайдеры контента (модули в `backend/providers`):**
    *   Каждый файл (например, `youtube.py`, `vk.py`, `hdrezka.py`, `torrent.py`) представляет собой отдельный модуль, реализующий специфическую логику взаимодействия с конкретной платформой. Они "знают", как искать контент и извлекать ссылки на стриминг со "своих" платформ.

## 3. Каким образом идет стриминг по разным платформам

KinoVibe сам по себе **не является платформой для хостинга или прямой трансляции видеоконтента**. Вместо этого он действует как **агрегатор ссылок на стриминг**. Процесс выглядит следующим образом:

1.  **Поиск ссылки:**
    *   Когда пользователь выбирает фильм или запрашивает стримы для конкретного фильма (через функцию `find_film_streams`), KinoVibe запускает `aggregate_search`.
    *   `aggregate_search` параллельно опрашивает все зарегистрированные провайдеры (YouTube, VK, RuTube, KodiK, HDRezka, Filmix, NovaTorrent, YTS).
    *   Каждый провайдер (реализующий `BaseProvider`) имеет метод `search()`, который возвращает список `SearchResult` объектов, содержащих, в том числе, URL, по которому можно получить поток или перейти к видео.

2.  **Извлечение информации о потоке (`get_stream`):**
    *   После того как `find_film_streams` собрала и отсортировала `SearchResult` от разных провайдеров, для каждого выбранного результата может быть вызван метод `get_stream(url)` соответствующего провайдера.
    *   Например, для `YouTubeProvider`, метод `get_stream` вызывает `yt-dlp` с определенными флагами (`-g`, `-f "best[ext=mp4]/best"`) для извлечения прямой `stream_url` из YouTube-видео.
    *   Для других провайдеров, таких как `HDRezkaProvider` или `KodikProvider`, `get_stream` будет реализован иначе: он может парсить веб-страницы, использовать их внутренние API или JS-инъекции для получения URL плеера или прямой ссылки на видеофайл.
    *   Для торрентов (`NovaTorrentProvider`, `YTSProvider`), `get_stream` может возвращать `magnet`-ссылки, которые затем обрабатываются внешними торрент-клиентами (например, Torrserver, упомянутый в экосистеме).

3.  **Стандартизация `StreamInfo`:**
    *   Результат извлечения потока стандартизирован в датакласс `StreamInfo`, который содержит:
        *   `stream_url`: Прямая ссылка на видеопоток.
        *   `provider`: Имя провайдера.
        *   `protocol`: Тип протокола (`http`, `hls`, `magnet`, `embed`). Это критично для фронтенда, чтобы выбрать правильный способ воспроизведения.
        *   Опционально: `audio_url`, `embed_url`.

4.  **Воспроизведение на фронтенде:**
    *   Полученные `StreamInfo` объекты (или их `stream_url`/`embed_url`) затем передаются клиентскому приложению (фронтенду).
    *   Фронтенд, основываясь на `protocol` и `stream_url`, использует соответствующий видеоплеер или механизм для воспроизведения контента. Например:
        *   `http`/`hls` протоколы: стандартный HTML5 `<video>` тег или специализированные плееры (например, Video.js, hls.js).
        *   `magnet` протокол: передается торрент-клиенту или используется для воспроизведения через Torrrserver.
        *   `embed` протокол: встраивается в iframe.

Таким образом, KinoVibe предоставляет **агрегированный каталог контента и механизмы для получения прямых ссылок** на видеопотоки, перекладывая саму функцию воспроизведения на клиентское приложение и внешние провайдеры.
